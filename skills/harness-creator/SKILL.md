# 하네스 크리에이터

사용자 목표에 맞는 가장 작은 유효 에이전트 팀 하네스를 만듭니다.

이 스킬의 목적은 메타 문서를 길게 작성하는 것이 아닙니다. 현재 프로젝트에서 바로 실행할 수 있는 팀 구조를 빠르게 만드는 것입니다.

## 기본 관점

이 스킬은 하네스를 아래 요소의 묶음으로 봅니다.

- 역할별 에이전트
- 팀 오케스트레이션 스킬
- 검증 역할 또는 검증 단계

즉 기본 산출물의 중심은 `문서`보다 `에이전트 팀`입니다.

## 작업 순서

반드시 이 순서를 따릅니다.

1. 사용자의 목표를 한두 줄로 다시 정리합니다.
2. 이 목표에 전용 하네스가 필요한지 판단합니다.
3. 필요하다면 목표를 지원할 수 있는 가장 작은 역할 세트를 고릅니다.
4. 역할 중 하나는 가능하면 검증 책임을 갖도록 설계합니다.
5. 초기 폴더와 파일 구조를 제안합니다.
6. 각 역할에 대응하는 `.codex/agents/<role>.toml` 초안을 작성합니다.
7. 팀을 오케스트레이션하는 `.agents/skills/<team-skill>/SKILL.md`를 작성합니다.
8. 루트 `AGENTS.md`에 구조, 사용법, 산출물 정의를 간단히 정리합니다.

전용 하네스가 필요 없다면 그렇게 명확히 말하고 더 단순한 경로를 제안합니다.

## 하네스 필요 여부 판단

아래 중 하나라도 참이면 하네스를 권장합니다.

- 사용자가 이 작업을 반복할 가능성이 있음
- 작업이 여러 역할의 협업으로 더 잘 풀림
- 도메인별 전문 역할을 분리하는 편이 품질에 유리함
- 검증, 리뷰, 압박 질문 같은 별도 시선이 필요함
- 하나의 지시문보다 팀 운영 규칙이 더 중요함

작업이 분명한 일회성이며 역할 분리의 이점이 거의 없으면 하네스를 생략합니다.

## 역할 선택

항상 최소 역할 세트를 우선합니다.

기본 패턴은 다음과 같습니다.

- `1개 역할`: 아주 단순한 선형 작업
- `2개 역할`: 생성 + 검증
- `3개 역할`: 계획 + 실행 + 검증
- `4개 이상`: 명확한 다단계 운영 필요가 있을 때만

검증은 가능한 한 기본 포함으로 봅니다. 최소한 아래 둘 중 하나는 있어야 합니다.

- 검증 전용 역할 하나
- 팀 스킬 안의 명시적 검증 단계 하나

## 기본 구조

다음 구조를 기본으로 삼고 필요할 때만 줄이거나 확장합니다.

```text
AGENTS.md
.agents/
  skills/
    <team-skill>/
      SKILL.md
.codex/
  agents/
    <role>.toml
```

역할별 에이전트는 `.codex/agents/` 아래에 둡니다.

팀 전체를 움직이는 오케스트레이션 스킬은 `.agents/skills/` 아래에 둡니다.

## 필수 출력

하네스를 제안하거나 생성할 때는 아래를 짧고 명확하게 제공합니다.

### 1. 하네스 유형

의도한 에이전트 팀 하네스를 한 줄로 설명합니다.

예시:

`서비스 런칭을 위한 3역할 검증 포함 에이전트 팀 하네스`

### 2. 팀 구성

초기 역할만 나열하고 역할마다 한 문장으로 설명합니다.

가능하면 각 역할의 책임은 분명히 갈라지게 합니다.

예시:

- `planner`: 목표를 단계와 의사결정 기준으로 정리
- `builder`: 실제 산출물 작성
- `validator`: 가정, 리스크, 근거 부족을 검증

### 3. 시작 구조

사용자가 처음 만들 디렉토리 트리를 보여줍니다.

### 4. 역할별 에이전트 초안

각 역할에 대응하는 `.codex/agents/<role>.toml` 초안을 작성합니다.

최소한 아래 필드를 포함합니다.

- `name`
- `description`
- `developer_instructions`

가능하면 아래도 함께 설계합니다.

- `model`
- `model_reasoning_effort`
- `sandbox_mode`

필요할 때는 아래도 추가할 수 있습니다.

- `[mcp_servers.<name>]`
- `url`

### 5. 팀 스킬 초안

`.agents/skills/<team-skill>/SKILL.md` 전체 초안을 작성합니다.

이 스킬은 개별 역할의 세부 규칙보다 팀 오케스트레이션에 집중해야 합니다.

설명문 frontmatter는 짧게 끝내지 말고, 아래를 자연스럽게 포함하는 긴 설명을 권장합니다.

- 이 스킬이 다루는 전체 범위
- 사용자가 어떤 표현으로 이 스킬을 호출할 수 있는지
- 기존 자산이 있을 때 어떤 부분만 부분 실행할 수 있는지
- 이 스킬의 범위 밖이 무엇인지

본문에는 가능하면 아래 섹션을 포함합니다.

- `실행 모드`
- `에이전트 구성`
- `워크플로우`
- `작업 규모별 모드`
- `데이터 전달 프로토콜`
- `에러 핸들링`
- `테스트 시나리오`
- `에이전트별 확장 스킬`

특히 워크플로우는 phase 단위와 작업 표를 함께 써서 의존 관계가 드러나게 작성합니다.

### 6. 루트 AGENTS.md 초안

루트 `AGENTS.md`는 가볍게 유지합니다.

가능하면 아래 3개 섹션만 포함합니다.

- `구조`
- `사용법`
- `산출물`

스타일은 팀 카탈로그처럼 보이게 작성합니다.

- 어떤 에이전트와 스킬이 있는지 한눈에 보이게 적습니다.
- 각 파일이 무슨 역할인지 짧게 설명합니다.
- 실제 결과물이 `_workspace/`에 어떤 이름으로 저장되는지 명시합니다.

## 작성 규칙

- 출력은 기본적으로 한국어로 작성합니다.
- 설명보다 바로 쓸 수 있는 파일 초안을 우선합니다.
- 에이전트 생성은 선택이 아니라 기본 산출물로 봅니다.
- 검증은 가능한 한 모든 도메인에 기본 포함으로 봅니다.
- 루트 `AGENTS.md`는 가볍게 유지합니다.
- 전역 플러그인 패키징을 기본값으로 두지 않습니다.
- 에이전트 `.toml`은 `name`, `description`, `developer_instructions`만 적는 최소형보다 실행 맥락까지 드러나는 풍부한 정의를 우선합니다.

다음은 피합니다.

- 긴 설계 배경 설명
- 역할은 많지만 책임이 겹치는 팀 구성
- 검증 없이 생성만 있는 팀 구성
- 실제 실행으로 이어지지 않는 문서 중심 출력
- 영어 위주의 기본 출력
