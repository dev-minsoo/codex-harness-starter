# Harness

사용자 목표에 맞는 가장 작은 유효 하네스를 만듭니다.

이 스킬의 목적은 큰 프레임워크를 설계하는 것이 아닙니다. 빈 디렉토리에서 실제로 실행 가능한 시작점까지 빠르게 이동하게 하는 것입니다.

## 작업 순서

반드시 이 순서를 따릅니다.

1. 사용자의 목표를 한두 줄로 다시 정리합니다.
2. 전용 하네스가 필요한지 판단합니다.
3. 필요하다면 그 목표를 지원할 수 있는 가장 작은 역할 세트를 고릅니다.
4. 초기 폴더와 파일 구조를 제안합니다.
5. 실제로 사용할 첫 `SKILL.md`를 작성합니다.
6. 다음 단계에서 바로 실행할 첫 프롬프트를 만듭니다.

전용 하네스가 필요 없다면 그렇게 명확히 말하고 더 단순한 경로를 제안합니다.

## 하네스 필요 여부 판단

아래 중 하나라도 참이면 하네스를 권장합니다.

- 사용자가 이 워크플로를 반복할 가능성이 있음
- 작업이 일관된 구조나 역할 분리의 이점을 가짐
- 재사용 가능한 프롬프트, 시작 파일, 플러그인 패키징이 필요함
- 계획에서 구현, 리서치에서 문서 작성, 구현에서 리뷰처럼 명확한 핸드오프가 있음

작업이 분명한 일회성이며 재사용이나 구조화의 이점이 없으면 하네스를 생략합니다.

## 역할 선택

항상 최소 역할 세트를 우선합니다.

기본 패턴은 다음과 같습니다.

- `단일 역할`: 선형 작업을 위한 `harness-creator` 하나 또는 도메인 스킬 하나
- `2개 역할`: 만들고 검토하는 흐름을 위한 `builder`와 `reviewer`
- `3개 역할`: 사전 분해가 도움이 되는 작업을 위한 `planner`, `builder`, `reviewer`

사용자가 더 강한 역할 분리를 명시적으로 원하지 않는 한 세 개를 넘기지 않습니다.

## 기본 구조

다음 구조를 기본으로 삼고 필요할 때만 줄이거나 확장합니다.

```text
.codex-plugin/
  plugin.json
AGENTS.md
skills/
  <primary-role>/
    SKILL.md
starter-prompts/
  <workflow>.md
```

하네스에 역할이 둘 이상이면 각 역할은 `skills/` 아래의 독립된 폴더에 둡니다.

## 필수 출력

하네스를 제안할 때는 아래 섹션을 짧고 명확하게 제공합니다.

### 1. 하네스 유형

의도한 하네스를 한 줄로 설명합니다.

예시:

`제품 요구사항을 실제 구현 작업으로 전환하는 2역할 PRD-to-delivery 하네스`

### 2. 역할

초기 역할만 나열하고 역할마다 한 문장으로 설명합니다.

### 3. 시작 구조

사용자가 처음 만들 디렉토리 트리를 보여줍니다.

### 4. 첫 스킬 초안

첫 `SKILL.md` 초안을 전체로 작성합니다. 사용자가 모든 역할을 한 번에 원하지 않는다면 우선 핵심 역할 하나만 작성합니다.

짧고 절차적이며 바로 쓸 수 있어야 합니다.

### 5. 첫 프롬프트

새 디렉토리에서 즉시 실행할 수 있는 프롬프트 하나를 작성합니다.

프롬프트에는 다음이 들어가야 합니다.

- 목표가 무엇인지
- Codex가 하네스를 사용하라고 지시하는 내용
- 추상적 계획이 아니라 실제 파일 생성이나 실행을 요청하는 내용

## 작성 규칙

긴 설명보다 짧은 실행 지시를 우선합니다.

다음은 피합니다.

- 긴 설계 배경 설명
- 큰 reference 섹션
- 막연한 미래 역할 제안
- 즉시 실행할 수 없는 문서 위주 출력

## 스타터 프롬프트 템플릿

사용자 워크플로와 맞는 경우 `starter-prompts/`를 기본 패턴의 출처로 사용합니다.

- `prd-to-delivery.md`
- `code-review.md`
- `research-to-write.md`

셋 중 잘 맞는 것이 없다면 억지로 맞추지 말고 사용자 맞춤 첫 프롬프트를 새로 작성합니다.
